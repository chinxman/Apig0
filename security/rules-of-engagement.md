# Rules Of Engagement

## Scope Enforcement

- Read `security/scope.md` before any pentesting-style validation.
- Test only assets marked authorized in `security/scope.md`.
- Refuse all targets not explicitly authorized.
- Do not infer authorization from public availability.

## Authorization Requirement

- Local repository, localhost, local Docker containers, and CI test environments are authorized for internal validation.
- Staging and production targets require written authorization in `security/scope.md`.
- Pentest-Swarm-AI-style automation requires an explicit in-scope target and evidence output path before execution.

## Safe Testing Rules

- Prefer SAST, dependency scanning, secret scanning, SBOM generation, unit tests, integration tests, configuration review, and baseline API checks.
- Use only non-destructive proof-of-concept validation.
- Keep request rates low and controlled.
- Stop if testing affects availability, logs unexpected errors, touches real customer data, or leaves the defined scope.

## Prohibited Testing

- No data destruction.
- No persistence.
- No third-party testing.
- No public target scanning.
- No brute forcing real accounts.
- No credential theft.
- No secret extraction.
- No data exfiltration.
- No malware.
- No backdoors.
- No disabling or bypassing logging.
- No stealth.

## Evidence Handling

- Record every command, timestamp, target, tool, operator, branch, commit, result, and output path.
- Store evidence under `security/evidence/` and reports under `security/reports/`.
- Redact accidental secrets from reports while preserving enough context for traceability.
- Imported artifacts must disclose origin, tooling attribution, validation status, and whether they came from `apig0-security-lab`.

## Cleanup Requirements

- Stop local services or containers started for validation unless the owner asks to keep them running.
- Remove temporary test data created during validation.
- Do not remove security evidence generated by the run.

## Reporting Requirements

- Document findings in `security/reports/findings.md`.
- Document remediation in `security/reports/remediation-plan.md`.
- Document residual risk in `security/reports/risk-register.md`.
- Update `security/readiness-stamp.md` honestly after checks complete.

## Stop Conditions

- A target is not in scope.
- Authorization is unclear.
- A tool would perform destructive or high-volume testing.
- Testing could affect third-party systems.
- Real secrets, production data, or customer data may be exposed.
- The repository owner requests a stop.
